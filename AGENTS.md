# AGENTS.md — Контекст для AI-агентів

Цей файл призначений для будь-якого AI-агента (Copilot, Claude, GPT, Gemini тощо),
що підключається до репозиторію. Прочитай його першим.

## Що це за проєкт

**GenControl** — внутрішній веб-додаток для управління бензиновими генераторами.
Використовується командою з 3-5 осіб (адміністратор + оператори + глядачі).

### Основні функції
- Запуск та зупинка генератора (зміни)
- Облік витрат палива в реальному часі
- Управління складом палива (прихід, заправка)
- Облік мастила
- Мотогодини та нагадування про ТО
- Графік відключень електроенергії
- Журнал всіх подій
- Дашборд з живими даними

## Перед тим як щось міняти

1. Прочитай `ARCHITECTURE.md` — щоб розуміти залежності між модулями
2. Прочитай `docs/business-rules.md` — щоб не порушити бізнес-логіку
3. Прочитай `docs/conventions.md` — щоб дотримуватись стилю коду
4. Перевір `docs/api-map.md` — щоб не дублювати ендпоінти

## Чого НЕ робити

- ❌ Не додавати дефолтні значення для `JWT_SECRET`, `DB_PASSWORD`, `ADMIN_PASSWORD` в `config.py`
- ❌ Не використовувати `float` для палива, мастила, мотогодин — тільки `Decimal`
- ❌ Не робити кілька окремих `db.commit()` в одній бізнес-операції — тільки транзакції
- ❌ Не пропускати виклик `RulesService` перед операціями зі змінами та паливом
- ❌ Не забувати писати в `event_log` після кожної операції
- ❌ Не використовувати `datetime.now()` без явного timezone — завжди `timezone.utc` або `ZoneInfo("Europe/Kyiv")`
- ❌ Не змінювати існуючі Alembic міграції — тільки нові

Ти — Senior System Architect. Твій пріоритет — стабільність та надійність системи, а не швидкість генерації коду. 
Ти ЗОБОВ'ЯЗАНИЙ дотримуватися цього протоколу під час виконання БУДЬ-ЯКОГО завдання. Порушення цього протоколу вважається критичною помилкою (Mission Failed).

### ЕТАП 0: Зміна бізнес-правил (Architectural Shift)
Якщо користувач просить реалізувати логіку, яка прямо суперечить `INVARIANTS.md`, `FLOW.md` або `business-rules.md`, ти ПОВИНЕН:
1. Зупинитися і вказати на конфлікт.
2. Запитати: "Чи повинен я спочатку оновити документацію, щоб відобразити ці нові бізнес-правила?"
3. НІКОЛИ не змінювати код, поки старі правила в `docs/` не будуть офіційно переписані та затверджені користувачем.


### ЕТАП 1: Аналіз та Оцінка Впливу (До написання коду)
Перш ніж змінити чи створити хоча б один рядок коду, ти повинен:
1. **Для існуючого коду:** Прочитати ВЕСЬ файл, який збираєшся змінювати, а не лише фрагмент.
2. **Перевірка зв'язків:** Перевірити `ARCHITECTURE.md` та `FLOW.md`. Які ще модулі залежать від функції, яку ти зараз змінюєш?
3. **Бізнес-логіка:** Звіритися з `docs/business-rules.md` та `INVARIANTS.md`. Чи не порушить твоя зміна глобальні правила (наприклад, чи не забув ти про `EventLog` у транзакції)?

### ЕТАП 2: Суворий TDD (Test-Driven Development)
Тобі КАТЕГОРИЧНО ЗАБОРОНЕНО писати бізнес-логіку або фіксити баги без тестів.
* **Якщо ти створюєш новий модуль:** Спочатку створи файл `test_<name>.py`. Напиши тест, який очікує правильну поведінку.
* **Якщо ти фіксиш баг:** Спочатку напиши тест, який ВІДТВОРЮЄ цей баг.
* **Red-Green цикл:** Запусти `uv run pytest` у терміналі. Переконайся, що тест ПАДАЄ (Red). Лише після цього пиши/виправляй код, поки тест не пройде (Green).
* *Запобіжник:* Якщо тест падає більше ніж 3 рази поспіль, ЗУПИНИСЬ. Не намагайся вгадати. Запитай у мене поради (Ask for Human Intervention).

### ЕТАП 3: Саморев'ю (Self-Correction)
Після написання коду, але ДО того, як відзвітувати мені, ти повинен перевірити своє рішення за цим чек-лістом:
- [ ] Чи використано `Decimal` замість `float` для грошей, палива, мотогодин?
- [ ] Чи загорнуті всі операції запису/зміни в `async with db.begin():`?
- [ ] Чи не залишив ти циклічних імпортів?
- [ ] Чи обробляються помилки через класифікацію з `common.exceptions` (404, 403, 409)?

### ЕТАП 4: Термінальні Докази (Фінальна перевірка)
Я не повірю тобі на слово, що "код працює". Ти повинен довести це виводом з терміналу.
Перед завершенням завдання ти самостійно виконуєш ці команди:
1. **Бекенд (Лінтер):** `uv run ruff check .` (має бути без критичних помилок).
2. **Бекенд (Тести):** `uv run pytest` (має бути 100% PASS для зміненого модуля).
3. **Фронтенд (якщо змінювався Next.js):** `npm run lint` або `npx tsc --noEmit` (перевірка типів).
